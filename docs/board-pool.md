# Board pool ÔÇö power banding, patches, and the regeneration lifecycle

Enemy boards are real, buildable `BoardSnapshot`s ÔÇö captured player/friend boards + house-bot boards ÔÇö
served back as opponents (`pickOpponent`). This doc defines how we measure their strength (**power
banding**), stamp them with a **patch**, **prune** stale ones, and **regenerate** a competitive pool when
the meta shifts.

## Power banding ÔÇö wave-relative strength

A board's strength only means something **relative to its wave**: a 7-wide board that's strong at wave 3 is
weak at wave 15. So `rateBoardForWave(board, wave, ladders)` (`packages/sim/src/rating.ts`) scores a board by
the **fraction of its OWN wave's calibration ladder it beats** (win = 1, draw = 0.5) Ôćĺ a 0..1 rating;
`ratingBand` then buckets it into `BAND_COUNT` (8) bands. **Band 0 = weak-for-the-wave, band 7 =
strong-for-the-wave.**

- **The ladder** (`buildWaveLadders`): per wave, a set of reference boards spanning weak Ôćĺ strong **current**
  play, generated by the smart bot (`buildBootstrapPool`) at a few fixed seeds ├Ś rising `fidelity` (0.2ÔÇŽ1.0).
  Real, synergy-aware boards ÔÇö not vanilla stat rungs. Deterministic. Because it's rebuilt from the live card
  set on every bake, **re-running per patch self-recalibrates.**
- **Why wave-relative:** the old `rateBoard` fought one fixed gauntlet (top rung 7/9/16) and **saturated** ÔÇö
  by ~wave 8 any decent board beat all rungs Ôćĺ rating `1.0`, so it couldn't tell a weak high-wave board from
  a strong one. That's exactly how high-wave boards silently went weak after a balance patch.
- **Curation / QA only.** Live matchmaking (`pickOpponent`) still matches by wave then biases to the closest
  `╬ú(atk+hp)` power. Bands drive the pool bake + pruning, **not** opponent selection (yet ÔÇö see the roadmap).

## Patches

Every board carries `patch` (`BoardSnapshot.patch`) = `<pkg version>+<short git sha>` (e.g. `0.1.0+a1b2c3d`),
stamped:

- at **capture** ÔÇö `boardLibrary.saveRunBoards`, via the Vite `__APP_VERSION__` / `__BUILD_SHA__` defines;
- at **house bake** ÔÇö `build-pool.ts`, via `package.json` version + `git rev-parse --short HEAD`.

**Convention: bump `package.json` `version` on each balance patch.** That version is the patch id, so boards
from before a balance change are identifiable + prunable. (Boards captured before stamping shipped have no
`patch` ÔÇö prune them with `--no-patch`.)

## Lifecycle / tooling

| Step | How |
|---|---|
| **Generate / capture** | House boards: `npm run pool` runs the bot. Your boards: finishing a run auto-captures per-wave boards to localStorage (`ascent.boards`); Export / Import via **Esc Ôćĺ Shared Boards**; drop exports into `docs/board-exports/` to commit them. |
| **Rate + band** | `npm run pool` builds the wave ladders, rates every board wave-relative, drops boards below the competitive floor, and prints a per-wave **band coverage** report (`wN:count[bMinÔÇôbMax]`). |
| **Prune stale** | `npm run pool:prune -- --before <YYYY-MM-DD>` / `--patch <id>` / `--no-patch` filters `docs/board-exports/*.json` (add `--dry-run` to preview); then `npm run pool` to re-bake. In-app: **Esc Ôćĺ Shared Boards Ôćĺ Clear my boards** wipes your localStorage captures. |
| **Regenerate (per patch)** | Bump `package.json` version Ôćĺ `npm run pool:prune -- --before <patch date>` (or `--no-patch`) Ôćĺ `npm run pool`. Tune the knobs in `build-pool.ts`: `HOUSE_SEEDS` (more boards), `FLOOR_BAND` / `FLOOR_FROM_WAVE` (how aggressively weak boards are dropped). |

### "Competitive enough to be fun"

The bake drops boards below band `FLOOR_BAND` (default `1` ÔÇö trivially weak for the wave) at waves Ôëą
`FLOOR_FROM_WAVE` (default `4`), so mid / late fights are never a free win, while early waves keep the full
range (winnable fights are good onboarding). The per-wave `count[band span]` line flags any wave that's gone
all-weak for the current meta ÔÇö the cue to regenerate or import stronger boards.

## Known limitation ÔÇö high-wave ladder ceiling

The calibration ladder is the **smart bot**, whose ceiling sits below expert human play, and at high waves
only strong (high-fidelity) bot runs survive ÔÇö so those ladders lack a *weak* end. The result: high-wave
ratings **compress toward band 7** (most real boards beat the bot's best), e.g. `w12:b7ÔÇôb7`. So the rating
discriminates well at low/mid waves but says little more than "beats the bot" past ~wave 9, and it can
*over*-rate high-wave boards relative to a skilled player. Until the ladder gets a higher ceiling (a stronger
bot, or curated strong reference boards), manage high-wave staleness with the **patch/date prune** + a
**re-bake** rather than the band floor. (Tracked in `roadmap.md`.)
